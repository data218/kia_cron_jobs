CURRENT ARCHITECTURE

Stack:
- Node.js
- Playwright
- PM2 for production process management
- Express webhook server for Android SMS Forwarder OTP
- dotenv
- node-cron
- Supabase JS SDK
- Direct PostgreSQL client for large JSONB append updates
- ExcelJS parser
- Pino structured logger
- Nodemailer failure alerts

Main runtime flow:
1. Login to KIA DMS using Playwright.
2. Use saved browser storage state when possible.
3. If login needs OTP, OTP is received through the local webhook server.
4. Run selected report modules in the authenticated browser session.
   - KIA DMS reports require KIA DMS login and OTP/session handling.
   - External reports such as RSA Report can skip KIA DMS login and open their own portal session.
5. Parse downloaded Excel files.
6. Store report data into Supabase table:
   public.business_excellence_am_kia_new

Supabase storage model:
- Current production direction is relational analytics tables plus JSON backup.
- The old JSON table remains optional raw backup/debug storage:
  public.business_excellence_am_kia_new
- New report uploads write the report-specific SQL table first.
- If the old JSON backup table is missing or unavailable, the report logs a warning and still succeeds after the relational save.
- Dedicated relational tables are created automatically from report sheet names.
- Current relational tables:
  ro_billing_report
  open_ro_yearly
  ew_report
  mcp_report
  rsa_report
  psf_yearly
  adv_wise_lubricants_vas
  operation_wise_analysis_report
  kia_call_center_complaints
- Relational table names and columns are normalized to lowercase snake_case.
- Each relational table includes:
  id bigserial primary key
  row_hash text unique
  uploaded_at timestamptz default now()
- Date-looking columns are stored as DATE.
- Amount/tax/count/rate/quantity/mileage style columns are stored as NUMERIC.
- Everything else is stored as TEXT.
- Inserts are batched and deduped with row_hash.
- Existing row_hash conflicts update the existing relational row instead of creating duplicates.
- Duplicate detection is based on the full normalized row values that are saved into SQL, excluding only system columns such as id, row_hash, and uploaded_at.
- Do not narrow row_hash to one or two business key columns; reports can have changed values under the same bill/RO/invoice/complaint number and those must be treated as distinct full-row records.
- Indexes are created automatically for row_hash, uploaded_at, date columns, bill_date, ro_date, service_advisor, service_type, and work_type when present.
- Migration command:
  npm run migrate:relational
- Migration can target one sheet:
  npm run migrate:relational -- "Kia call center complaints"

Legacy JSON backup model:
- One row per report/sheet.
- brand = kia
- sheet_name identifies the report dataset.
- headers stores JSON array of column headers.
- rows stores JSON array of row objects.
- uploaded_at is refreshed on every successful update.
- Existing sheet rows are merged by brand + sheet_name.
- Existing data must not be removed or replaced during normal report uploads.
- Incoming rows are compared against existing rows using a stable row signature.
- Only rows that are not already present are appended.
- Duplicate incoming rows are skipped.
- Headers are preserved and extended with any new incoming columns.
- Existing sheet updates append only new rows server-side through PostgreSQL when a row already exists.
- This avoids resending very large historical JSONB arrays through the Supabase REST API.
- DATABASE_URL is required for existing-sheet append updates.

Important report selection setting:
- REPORTS_TO_RUN

Current development/testing mode:
- REPORTS_TO_RUN defaults to all in production.
- Regular hourly scheduler mode excludes open-ro-yearly even when REPORTS_TO_RUN=all.
- Open RO Yearly has its own schedule lane.
- Production mode should use REPORTS_TO_RUN=all and TEST_SINGLE_REPORT=false.
- Temporary single-report mode is also supported with:
  TEST_SINGLE_REPORT=true
  TEST_REPORT_NAME=<report-id>
- Dry-run sequencing test is supported with:
  DRY_RUN_REPORTS=true
  DRY_RUN_REPORT_DELAY_MS=500
- Use npm run test:sequence to verify sequential order and lock behavior without browser login, downloads, or Supabase writes.

Cron schedules:
- Regular reports:
  REGULAR_REPORTS_CRON_SCHEDULE=0 10-18 * * *
  Runs hourly from 10:00 AM through 6:00 PM.
- Open RO Yearly:
  OPEN_RO_YEARLY_CRON_SCHEDULE=0 18 * * *
  Runs only once per day at 6:00 PM.
- Kia Call Center Complaints:
  KIA_CALL_CENTER_COMPLAINTS_CRON_SCHEDULE=0 18 * * *
  Runs only once per day at 6:00 PM.
- Scheduler uses one global in-process lock.
- If any lane is still running when another tick fires, the new tick is skipped.

To restore all reports later:
- Set REPORTS_TO_RUN=all

To run selected reports only:
- Use comma-separated report ids.
- Example:
  REPORTS_TO_RUN=ro-billing,kia-call-center-complaints,open-ro-yearly,psf-yearly,ew-report,mcp-report,adv-wise-lubricants-vas,rsa-report


PAGINATION / LOCAL EXPORT POLICY

All report modules should use the shared paged export flow when exporting Kendo grids.

Shared helper:
- src/reports/paged-export.js

Behavior:
1. Detect visible Kendo pager.
2. Export page 1.
3. If more pages exist, move through each page and export every page separately.
4. Save page exports temporarily under downloads/report-chunks/.
5. Merge and parse with ExcelJS.
5. Parse and merge all exported Excel files.
6. Normalize headers by preserving first-seen header order.
7. Upsert the merged dataset into Supabase.
8. Delete local export folder only after Supabase upload succeeds.
9. If upload fails, keep local export files for debugging.

Delay policy:
- Intentional per-step sleeps/waits should be 5000ms or less.
- The config helper caps report post-search/page-load/chunk delays at 5000ms even if a higher value is placed in .env.
- Long selector/network/login/captcha timeouts may remain higher because they are failure guards or manual-solve windows, not intentional pacing delays.

Production retry/alert policy:
- Each report is executed through src/utils/execute-with-retry.js.
- Default retries: 3.
- Default retry delay: random 30-60 seconds.
- A screenshot is captured to logs/screenshots/ on each failed attempt when a Playwright page is available.
- Failure email is sent only after all retry attempts fail.
- A failed report is logged and the master scheduler continues with remaining reports.
- Health status is written to logs/health.json.

Dashboard materialized view refresh:
- After a report lane finishes with zero failed imports, src/cron/scheduler.js refreshes dashboard materialized views before marking the job complete.
- Refresh helper: src/supabase/materialized-views.js.
- Views refreshed, in order:
  - workshop_performance_jc_summary_v1
  - workshop_operation_addon_summary_v1
- Refresh SQL uses REFRESH MATERIALIZED VIEW CONCURRENTLY and runs through a standalone Postgres client without an explicit transaction.
- If any import report fails, the view refresh is skipped and the scheduler logs the skipped reason.
- If a refresh fails, the helper logs the failed view and rethrows so the scheduler/PM2 logs the job failure.
- Dry-run report lanes skip view refresh because no source-table imports were committed.

Wi-Fi / network recovery:
- src/utils/network.js checks connectivity using NETWORK_CHECK_URL.
- Scheduler waits for connectivity before starting a lane.
- Each report waits for connectivity before attempting work.

- Playwright browser contexts set default action and navigation timeouts from PLAYWRIGHT_ACTION_TIMEOUT_MS and PLAYWRIGHT_NAVIGATION_TIMEOUT_MS.

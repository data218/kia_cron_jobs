CURRENT ARCHITECTURE

Stack:
- Node.js
- Playwright
- PM2 for production process management
- Express webhook server for Android SMS Forwarder OTP
- dotenv
- node-cron
- Supabase JS SDK
- Direct PostgreSQL client for large JSONB append updates
- ExcelJS parser
- Pino structured logger
- Nodemailer failure alerts

Main runtime flow:
1. Login to KIA DMS using Playwright.
2. Use saved browser storage state when possible.
3. If login needs OTP, OTP is received through the local webhook server.
4. Run selected report modules in the authenticated browser session.
   - KIA DMS reports require KIA DMS login and OTP/session handling.
   - External reports such as RSA Report can skip KIA DMS login and open their own portal session.
5. Parse downloaded Excel files.
6. Store report data into Supabase table:
   public.business_excellence_am_kia_new

Supabase storage model:
- One row per report/sheet.
- brand = kia
- sheet_name identifies the report dataset.
- headers stores JSON array of column headers.
- rows stores JSON array of row objects.
- uploaded_at is refreshed on every successful update.
- Existing sheet rows are merged by brand + sheet_name.
- Existing data must not be removed or replaced during normal report uploads.
- Incoming rows are compared against existing rows using a stable row signature.
- Only rows that are not already present are appended.
- Duplicate incoming rows are skipped.
- Headers are preserved and extended with any new incoming columns.
- Existing sheet updates append only new rows server-side through PostgreSQL when a row already exists.
- This avoids resending very large historical JSONB arrays through the Supabase REST API.
- DATABASE_URL is required for existing-sheet append updates.

Important report selection setting:
- REPORTS_TO_RUN

Current development/testing mode:
- REPORTS_TO_RUN defaults to all in production.
- Regular hourly scheduler mode excludes open-ro-yearly even when REPORTS_TO_RUN=all.
- Open RO Yearly has its own schedule lane.
- Production mode should use REPORTS_TO_RUN=all and TEST_SINGLE_REPORT=false.
- Temporary single-report mode is also supported with:
  TEST_SINGLE_REPORT=true
  TEST_REPORT_NAME=<report-id>
- Dry-run sequencing test is supported with:
  DRY_RUN_REPORTS=true
  DRY_RUN_REPORT_DELAY_MS=500
- Use npm run test:sequence to verify sequential order and lock behavior without browser login, downloads, or Supabase writes.

Cron schedules:
- Regular reports:
  REGULAR_REPORTS_CRON_SCHEDULE=0 10-18 * * *
  Runs hourly from 10:00 AM through 6:00 PM.
- Open RO Yearly:
  OPEN_RO_YEARLY_CRON_SCHEDULE=0 10,18 * * *
  Runs only at 10:00 AM and 6:00 PM.
- Scheduler uses one global in-process lock.
- If any lane is still running when another tick fires, the new tick is skipped.

To restore all reports later:
- Set REPORTS_TO_RUN=all

To run selected reports only:
- Use comma-separated report ids.
- Example:
  REPORTS_TO_RUN=ro-billing,kia-call-center-complaints,open-ro-yearly,psf-yearly,ew-report,mcp-report,adv-wise-lubricants-vas,rsa-report


PAGINATION / LOCAL EXPORT POLICY

All report modules should use the shared paged export flow when exporting Kendo grids.

Shared helper:
- src/reports/paged-export.js

Behavior:
1. Detect visible Kendo pager.
2. Export page 1.
3. If more pages exist, move through each page and export every page separately.
4. Save page exports temporarily under downloads/report-chunks/.
5. Merge and parse with ExcelJS.
5. Parse and merge all exported Excel files.
6. Normalize headers by preserving first-seen header order.
7. Upsert the merged dataset into Supabase.
8. Delete local export folder only after Supabase upload succeeds.
9. If upload fails, keep local export files for debugging.

8. RSA Report

Report id:
- rsa-report

Sheet name:
- RSA Report

Portal:
- https://kia.awpassistance.in/report

Login:
- Uses RSA_USER_ID and RSA_PASSWORD from .env.
- No OTP is required.
- Uses separate browser storage state:
  ./storage/rsa-portal-state.json
- When only rsa-report is selected, scheduler skips KIA DMS login entirely.
- If the portal shows a concurrent session warning, search for any visible Logout from everywhere button, click it, wait for the modal to close, then click Login again.
- Concurrent-session handling should be button-first because the modal wrapper/text can vary between RSA portal builds.

Navigation:
- After login, let the RSA dashboard load.
- Click the Report link/button from the dashboard.
- Do not force /report immediately after concurrent-session logout because the app returns to login first.
- Prioritize dashboard/header selectors:
  a[href="/report"]:has-text("Report")
  a[text-content="Report"]
  nx-header-navigation a:has-text("Report")
- Do not treat generic inputs as proof that the report interface loaded.
- The RSA report page is considered ready only when .nx-dropdown__container is visible.

Report type:
- Dropdown container: .nx-dropdown__container
- Select option: Policy Sale Detailed Report

Date fields:
- Start date: #nx-input-0
- End date: #nx-input-1
- After selecting Policy Sale Detailed Report, the portal does a small async reload before date fields are usable.
- Wait for loaders/network plus RSA_REPORT_PAGE_LOAD_DELAY_MS before filling dates.
- If fixed nx-input ids are not visible, fall back to the first two visible non-login inputs on the report filter form.

Date format:
- MM/DD/YYYY

Date logic:
- Start date = first day of current month.
- End date = current execution date.
- Example for 2026-05-21:
  05/01/2026 to 05/21/2026

Search:
- Click GET DETAILS.
- Wait for network/loaders.
- Additional wait comes from RSA_REPORT_POST_SEARCH_DELAY_MS because the table loads asynchronously.
- RSA flow uses human-like random pauses and slow typing to reduce captcha triggers.
- Timing config:
  RSA_HUMAN_DELAY_MIN_MS
  RSA_HUMAN_DELAY_MAX_MS
  RSA_TYPING_DELAY_MS
- Captcha handling is a manual pause, not a bypass.
- If a visible reCAPTCHA image challenge appears, the automation waits for the user to solve it and then continues.
- Captcha wait config:
  RSA_CAPTCHA_TIMEOUT_MS
- RSA can run with a reusable browser identity:
  RSA_USE_PERSISTENT_PROFILE=true
  RSA_USER_DATA_DIR=./storage/rsa-chrome-profile
- RSA can also attach to an already-running Chrome if Chrome is started with remote debugging and RSA_CDP_ENDPOINT is set.
- When attached through CDP, the automation leaves the active Chrome session open after the run.

Pagination:
- Uses RSA-specific generic pagination handling, not Kendo.
- Exports every page separately with the Export button.

DEVELOPMENT NOTES

When adding a new report:
1. Add navigation helper in src/navigation/kia-menu.js if needed.
2. Add a report module under src/reports/.
3. Register the report in src/reports/index.js with a stable id.
4. Add config values in src/config.js and update .env for active testing.
5. Store results through the existing Supabase report-store flow unless explicitly requested otherwise.
6. Run npm run check.
7. Update this PROJECT_CONTEXT.txt file with:
   - report id
   - sheet name
   - navigation path
   - selectors
   - date logic
   - storage behavior
   - status

Current testing preference:
- While remaining report crons are being built, run only the report currently under development.
- Avoid forcing all completed reports to run sequentially during development.
- Restore sequential all-report behavior after all remaining crons are built and tested.
- Use TEST_SINGLE_REPORT=true and TEST_REPORT_NAME=<report-id> for this; do not change production cron lanes.

Delay policy:
- Intentional per-step sleeps/waits should be 5000ms or less.
- The config helper caps report post-search/page-load/chunk delays at 5000ms even if a higher value is placed in .env.
- Long selector/network/login/captcha timeouts may remain higher because they are failure guards or manual-solve windows, not intentional pacing delays.

Production retry/alert policy:
- Each report is executed through src/utils/execute-with-retry.js.
- Default retries: 3.
- Default retry delay: random 30-60 seconds.
- A screenshot is captured to logs/screenshots/ on each failed attempt when a Playwright page is available.
- Failure email is sent only after all retry attempts fail.
- A failed report is logged and the master scheduler continues with remaining reports.
- Health status is written to logs/health.json.

Scheduler sequence test:
- src/cron/sequential-test.js
- npm run test:sequence
- Runs regular lane in dry-run mode.
- Runs Open RO Yearly lane after regular lane completes.
- Starts an overlapping Open RO request while regular lane is running and expects it to be skipped by the global lock.

Local PM2 status:
- PM2 process name: kia-cron-job
- PM2 config file: ecosystem.config.cjs
- Local PM2 setup was started and saved with pm2 save.
- PM2 runs src/cron/scheduler.js from C:\Users\HP\Downloads\Kia_Cron_Job.
- Scheduler entrypoint detection uses resolved filesystem paths so it works under PM2 on Windows and under Node directly.
- PM2 passes --scheduler explicitly so the scheduler lane registration is not dependent on PM2 argv path behavior.

Wi-Fi / network recovery:
- src/utils/network.js checks connectivity using NETWORK_CHECK_URL.
- Scheduler waits for connectivity before starting a lane.
- Each report waits for connectivity before attempting work.
- Each retry attempt waits for connectivity before running.
- Playwright browser contexts set default action and navigation timeouts from PLAYWRIGHT_ACTION_TIMEOUT_MS and PLAYWRIGHT_NAVIGATION_TIMEOUT_MS.
- This prevents a dropped Wi-Fi connection from leaving the automation stuck forever on a loading page; it should fail, wait for network recovery, and retry.
